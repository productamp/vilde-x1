I’d implement Phase 1 in three parts, using the existing seams in the app rather than inventing a second control path.

First, add a single persisted `productVibeMode` flag in renderer state and mirror it into a small main-process settings file. The renderer already uses Jotai `atomWithStorage` for preferences and onboarding state in `app/src/renderer/lib/atoms/index.ts`, but the main process decides whether to load the app or `login.html` before React boots in `app/src/main/windows/main.ts`. So the flag cannot live only in `localStorage`. I’d add a minimal main-side store like `app-mode.json`, expose `app:get-product-vibe-mode` and `app:set-product-vibe-mode` through preload in `app/src/preload/index.ts`, and have a `productVibeModeAtom` read/write through that.

Second, add a dev-only toggle in the hidden developer surface instead of the normal UI. The clean place is the existing Debug/Beta settings path in `app/src/renderer/features/settings/settings-sidebar.tsx` and `app/src/renderer/components/dialogs/settings-tabs/agents-debug-tab.tsx`. That gives us a way to switch between normal 1Code and ProductVibe mode without exposing the control to end users.

Third, branch auth and onboarding off that flag. In `app/src/main/windows/main.ts`, if `productVibeMode` is on, I’d bypass the `AuthManager.isAuthenticated()` gate and always load the app shell instead of `login.html`. I’d also make `auth:get-user` and `auth:is-authenticated` return ProductVibe-safe values when the mode is on, so renderer code doesn’t break. Then in `app/src/renderer/App.tsx`, I’d skip the account-level onboarding path and go straight to project selection or the main layout. I would not remove provider setup yet: Claude/Codex/API-key onboarding is still needed for actual generation, but 21st account login/signup would no longer block entry.

One caveat: removing the login wall is not enough by itself, because some UI still fires auth-backed remote queries after load, for example in `app/src/renderer/features/agents/ui/agents-content.tsx`. As part of Phase 1, I’d gate those queries behind `!productVibeMode` or make them no-op in ProductVibe mode, otherwise the app would boot but still throw auth-related runtime failures.
