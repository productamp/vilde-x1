I’d implement Phase 2 as a local preview system layered onto the existing UI, not as a new preview product. The app already has a mature iframe-based preview component in `app/src/renderer/features/agents/ui/agent-preview.tsx`, and it already has a unified right sidebar in `app/src/renderer/features/details-sidebar/details-sidebar.tsx`. So the plan would be to replace the preview source, not rebuild preview from scratch.

First, add a tiny static web server in Electron main process that serves the active local project/worktree directory over `127.0.0.1` on an ephemeral port. The right place is a new main-side helper, something like `app/src/main/lib/preview/preview-server.ts`, because preview server lifecycle belongs with other desktop-only runtime services. The server should do only the minimum needed for static sites: serve `/` as `index.html`, serve CSS/JS/images/fonts by relative path, block path traversal outside the workspace root, and allow simple clean-url fallback like `/about` -> `/about.html`. This is the smallest correct implementation for ProductVibe’s HTML-first phase.

Second, expose that local preview target to the renderer through desktop tRPC, keyed by chat ID. The chat already knows its `worktreePath` in the local desktop path, and that is enough to derive preview. I’d add a small `preview.ensureForChat` query under `app/src/main/lib/trpc/routers/preview.ts`, register it in the main app router, and have it return `{ baseUrl, port, rootPath }`. I would not add database schema for preview in Phase 2 because it is derivable runtime state from the workspace path.

Third, switch preview consumption in the renderer from sandbox-specific props to a generic `previewBaseUrl`. Right now `agent-preview.tsx` still assumes `sandboxId + port -> cloud URL`. I’d refactor it so it just takes a base URL string and keeps the rest of the behavior the same: iframe, responsive controls, reload button, path bar, path persistence. Then in local desktop chat views, `active-chat.tsx` and `agents-content.tsx` would call `preview.ensureForChat` when `worktreePath` exists and pass the returned localhost URL into `AgentPreview`. Remote sandbox preview can remain as fallback so this change doesn’t break older non-local flows.

Fourth, make preview visible in the right place instead of forcing a separate preview mode. Since the app already has a unified right panel with `Details` and `Files` tabs, I’d add a `Preview` tab there in `app/src/renderer/features/details-sidebar/atoms/index.ts` and `app/src/renderer/features/details-sidebar/details-sidebar.tsx`. That keeps the UI simple: chat in the center, project tools on the right, website preview as just another panel. In ProductVibe mode, I’d auto-open the unified right sidebar to `Preview` when local preview is available, and suppress the older standalone preview sidebar if unified sidebar mode is on.

Finally, wire preview refresh to local file changes. The existing desktop app already listens to file changes and git watcher events through `useFileChangeListener` and `useGitWatcher`. I’d reuse those signals in `active-chat.tsx` and `agents-content.tsx` to dispatch a lightweight `agent-preview-reload` event whenever the workspace changes. That is enough for Phase 2. No hot-module protocol, no websocket reload server, just iframe refresh when Claude edits files.

One caveat: localhost preview inside Electron will be blocked unless renderer CSP allows it. So as part of Phase 2, I’d update `app/src/renderer/index.html` to allow `http://127.0.0.1:*` and `http://localhost:*` in both `connect-src` and `frame-src`. Without that, the preview server can run successfully but the iframe will still stay blank.
